                                                            # maven

                                                            1. 点击刷新按钮,无法更新仓库jar包时，删除maven库中有问题的jar包，重新刷新拉取依赖包
                                                            2. 编译时如果发现代码没有生效考虑先删除编译输出文件即点击clean,然后执行编译命令
                                                            3. maven3.5之后，出现无法拉取依赖包时，考虑将依赖库中_remote.repositories文件删除
                                                            4. dependencies中的jar直接加到项目中，管理的是依赖关系（如果有父pom,子pom,则子pom中只能被动接受父类的版本）；dependencyManagement主要管理版本，对于子类继承同一个父类是很有用的，集中管理依赖版本不添加依赖关系，对于其中定义的版本，子pom不一定要继承父pom所定义的版本。参考链接open in new window
                                                            5. 依赖版本冲突 第一种：同名的class文件 解决办法：将优先级高的依赖包放在前面，根据依赖的加载顺序，是从上到下加载的 第二种：引入了重复的jar 典型错误：compatible versions of the classes ma.glasnost.orika.impl.MapperFacadeImpl and com.googlecode.concurrentlinkedhashmap.ConcurrentLinkedHashMap$Builder 解决办法：进行包依赖分析，找到引用重复包的位置，将包排除
                                                            <dependency>
			<groupId>com.alibaba.cloud</groupId>
			<artifactId>spring-cloud-starter-alibaba-sentinel</artifactId>
			<exclusions>
				<exclusion>
					<groupId>com.googlecode.concurrentlinkedhashmap</groupId>
					<artifactId>concurrentlinkedhashmap-lru</artifactId>
				</exclusion>
			</exclusions>
		</dependency>

                                                            # spring

                                                            1、问题

                                                            自己搭建的springboot项目启动报错，找不到相关配置信息，明明配置在bootstrap.yml中，以前一直用的application.yml，想着application.yml优先级没有bootstrap.yml高，bootstrap配置文件肯定没问题，结果项目就是死活起不起来。

                                                            原因

                                                            SpringBoot项目，只会识别application.* 配置文件，并不会自动识别bootstrap.yml。 bootstrap.yml配置是SpringCloud项目才会用到的，如果你想在springboot项目中用bootstrap.yml，那么你需要添加bootstrap启动器

                                                            添加bootstrap启动器

                                                            <dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-bootstrap</artifactId>
</dependency>

                                                            优先级问题

                                                            由于SpringCloud是基于SpringBoot构建的，所有SpringCloud项目两种文件都会识别，这个时候才有优先级的说法，SpringCloud项目是会优先读取bootstrap配置在读取application配置。 SpringCloud应用是基于bootstrap的上下文运行的。

                                                            # 问题2、解决 required a single bean, but 2 were found的spring注入bean错误

                                                            原因

                                                            IOC容器中有两个bean 分别是 ITestServiceImplOne、ITestServiceImplTwo，在调用的abstract类中，注入的为interface类，由于是泛型，并没有指向具体的实例bean，所以导致Autowired注入时不知道注入哪一个

                                                            解决方案

                                                            方式一 优先级使用

                                                            使用@Primary注解

                                                            方式二 默认值使用

                                                            引用具体的实现类加上@Qualifier

                                                            # 问题3、因为网络波动或其他原因导致接口调用失败，等待过后处理成功

                                                            解决方案 重处理即重试机制

                                                            常见的场景：

                                                            • 发送消息失败
                                                            • 调用远程服务失败
                                                            • 争抢锁失败

                                                            方式一 try/catch或者while循环进行重试

                                                            方式二 引入spring-retry重试依赖

                                                            <dependencys>
   <dependency>
      <groupId>org.springframework.retry</groupId>
      <artifactId>spring-retry</artifactId>
   </dependency>
   <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-aop</artifactId>
   </dependency>
</dependencys>

                                                            启用@Retryable

                                                            @EnableRetry
@SpringBootApplication
public class HelloApplication {

    public static void main(String[] args) {
        SpringApplication.run(HelloApplication.class, args);
    }

}

                                                            在方法上添加@Retryable

                                                                @Retryable(value = RetryException.class, maxAttempts = 4, backoff = @Backoff(delay = 1000, multiplier = 1.5), recover = "recover")

                                                            重试失败后调用处理方法，在方法上加上注解@Recover

                                                            # spring security

                                                            # 问题1：启用spring security后，post方式下一直报403错误

                                                            现象:启用spring security后，post方式下一直报403错误

                                                            原因：Spring Security为防止CSRF(Cross-site requetst forgery跨站请求伪造)的发生，限制了除了get以外的大多数方法。

                                                            解决方案1：

                                                            屏蔽CSRF控制，即Spring Security不再限制CSRF,进行配置：

                                                            @Override
protected void configure(HttpSecurity http) throws Exception {
 	//屏蔽CSRF控制
 	http.csrf().disable()
	... 
}

                                                            解决方案②：

                                                            定义headers，post方式提交的时候带上headers的信息：

                                                            var headers = {};
headers['X-CSRF-TOKEN'] = "[[${_csrf.token}]]";
$.ajax({
    url: url,
    type: "POST",
    headers: headers,
    dataType: "json",
    success: function(result) {
    }
});

                                                            解决方案③：

                                                            直接作为参数提交:

                                                            $.ajax({
    url: url,
    data: {
        "[[${_csrf.parameterName}]]": "[[${_csrf.token}]]"
        },
    type: "POST",
    dataType: "json",
    success: function(result) {
    }
});

                                                            解决方案④：

                                                            form表单提交的时候，作为隐藏参数提交

                                                            <input type="hidden" th:name="${_csrf.parameterName}" 
	th:value="${_csrf.token}">

                                                            # 问题2：权限配置不生效问题

                                                            现象：权限配置不生效问题

                                                            原因：没有开启基于方法的权限认证

                                                            解决办法: 在配置类上加@EnableGlobalMethodSecurity(prePostEnabled = true)

                                                            # nacos注册服务 ip 是本地虚拟机ip 而不是本地机器ip

                                                            原因： 和Naocs/Consul无关。默认情况下，Spring Cloud注册的是第一块非回环网卡地址所对应的网卡，详见 org.springframework.cloud.commons.util.InetUtils#findFirstNonLoopbackHostInfo 。这意味着如果第一块非回环地址对应的网卡恰好是你那个虚拟网卡，也照样会注册 解决办法：

                                                            # currentServerAddr:http://localhost:8848, err : Connection refused (Connection refused)

                                                            原因是：你引入了config依赖但是不配置nacos配置的地址：他就默认为localhost:8848这个服务，正好我本地启动了这个服务，配置参数和服务器上是一致的，所以开发时能用 解决办法：添加nacos的配置地址

                                                            spring.cloud.nacos.config.server-addr=http://192.168.158.128:8848
spring.cloud.nacos.config.enabled=true

                                                            # Param ‘serviceName‘ is illegal, serviceName is blank

                                                            serviceName 来自于配置文件，如果没有指定，默认使用${spring.application.name:} 如果spring.application.name 没有指定 就默认为""; 原因： bootstrap没有被加载，spring-cloud-dependencies 2020.0.0 版本不在默认加载bootstrap 文件 解决办法： 加载bootstrap 文件需要手动添加依赖

                                                                    <dependency>
            <groupId>org.springframework.cloud</groupId>
            <artifactId>spring-cloud-starter-bootstrap</artifactId>
        </dependency>

                                                            # 整合引入sentinel 报循环依赖

                                                            解决办法：升级cloud版本，版本参考Alibaba官方文档，高版本解决了循环引入的问题

                                                            # 引入gateway网关依赖时无法正常启动

                                                            原因：与spring-boot-starter-web依赖冲突 解决办法： 删除spring-boot-starter-web依赖

                                                            # 拉取nacos配置是一直报错，应该访问配置中心的地址，但是确访问本地地址

                                                            报错信息：[NACOS SocketTimeoutException httpGet] currentServerAddr:http://localhost:8848， err : connect timed out 原因：bootstrap.yml配置文件应为缓存等原因无法加载，加载的过程中认为配置文件不存在 解决办法：将bootstrap.yml文件删除,创建一个新的配置文件

                                                            # 其他

                                                            # 1、java.lang.UnsupportedOperationException: null at com.google.common.collect.ImmutableCollection.add

                                                            使用com.google.common.collect.ImmutableList初始化List，当进行add、remove操作时，报java.lang.UnsupportedOperationException: null 错误。

                                                            原因：

                                                            不是所有初始化的List都有add remove等操作。

                                                            例如Arrays.asList和一些第三方工具比如google ImmutableList初始化返回的List。

                                                            调用Arrays.asList()生产的List的add、remove方法时报异常，这是由Arrays.asList() 返回的市Arrays的内部类ArrayList， 而不是java.util.ArrayList。Arrays的内部类ArrayList和java.util.ArrayList都是继承AbstractList，remove、add等方法AbstractList中是默认throw UnsupportedOperationException而且不作任何操作。java.util.ArrayList重新了这些方法而Arrays的内部类ArrayList没有重新，所以会抛出异常。

                                                            ImmutableList是不可变的对象

                                                            ImmutableList抽象类add方法：

                                                            解决办法： 使用原生的创建list的对象的方式

                                                            List<Integer> list = new ArrayList<>(ImmutableList.of(11, 22, 33));
list.add(555);

                                                            # 2、org.xml.sax.SAXParseException； 对实体 “characterEncoding“ 的引用必须以 ‘；‘ （xml的转义字符）

                                                            场景： 原因：在xml文件中有以下几类字符要进行转义 解决办法：将&改成&

                                                            # 3、github pages 无法加载css或js,带下划线的文件里面的内容都无法访问

                                                            场景：github pages 不能识别下划线开头的文件

                                                            原因：github pages的jekyll模版会忽略下划线开头的文件，所以要禁用jekyll

                                                            解决办法：在项目根目录下添加一个空白文件，命名为：.nojekyll

                                                            # 4、整合流程引擎时报Field transactionManager in org.camunda.bpm.spring.boot.starter.configuration.impl.DefaultDatasourceConfiguration

                                                            原因：没有指定事务管理器，初始化流程引擎表时需要引入mybatis依赖 解决办法：引入mybatis依赖

                                                            # 5、部署流程图实例成功，结果响应信息报错

                                                            原因：结果响应时，fastjson无法对xml结构的信息做处理 解决办法：只能将部署成功后返回的对象做处理，比如返回部署实列Id

                                                            # 浏览器localstorage或cookie中数据被莫名其妙清除

                                                            原因：时区问题，时间转换之后导致数据过期时间到了被浏览器强制删除

                                                            # 系统安全

                                                            # crsf跨站请求伪造

                                                            解决办法： 浏览器设置同源策略(是浏览器的一种安全策略)，即页面所在服务器地址和资源服务器地址必须协议、地址、端口号保持一致

                                                            # 同源策略引发的跨域问题

                                                            解决办法：

                                                            1. nginx做反向代理，保证页面资源地址和代理服务器地址一直
                                                            2. 放开服务端跨域限制即cors是跨域资源共享，通过添加response的header的方法（参考header('Access-Control-Allow-Origin:*');
                                                            3. SONP的方式，JSONP方式的原理是将返回数据以资源的方式放在script标签里面返回，由于返回的不是JSON数据，就没有跨域的限制了，类似对页面添加的js引用，这样就解决了跨域的问题。但需要在服务端拼接结构，违背了restful的原则
